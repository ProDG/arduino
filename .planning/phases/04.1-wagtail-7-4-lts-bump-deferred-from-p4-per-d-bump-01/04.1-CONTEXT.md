<decisions>
## Implementation Decisions

### Pin & Lock

- **D-PIN-01:** Pin specifier becomes `wagtail~=7.4.0` (compatible-release operator, allows 7.4.x patch updates within LTS series). Mirrors existing `~=7.3.0` pattern. NOT `==7.4.0` (denies patches), NOT `>=7.4,<7.5` (more permissive than the project's existing convention).
- **D-PIN-02:** Lock refresh uses `uv lock --upgrade-package wagtail` (targeted), not `uv lock --upgrade`. Transitive bumps caused by Wagtail's own constraints are acceptable; unrelated dep churn is not.
- **D-PIN-03:** `wagtail-headless-preview~=0.8` stays as-is unless a verification gate fails specifically because of it. If a re-pin is required, take the minimum bump that restores compatibility and document the reason in the PLAN summary. If no compatible release exists within ~1 day of effort → invoke D-BUMP-03 rollback.
- **D-PIN-04:** Other Wagtail-adjacent pins (`django-storages[s3]`, `boto3`, `psycopg`, `Django 5.2 LTS`, `Python 3.13`) are NOT touched in this phase. They were locked in P4 and have no relationship to the 7.3→7.4 minor bump.

### Verification (D-BUMP-02 gates — locked, listed here for executor reference)

- **D-VERIFY-01:** Five gates, in this order, each as its own atomic commit:
  1. **Pin + lock + build** — `pyproject.toml` edit, `uv.lock` refresh, `docker compose build wagtail` succeeds, `docker compose up -d` brings the stack healthy.
  2. **Wagtail admin smoke** — `curl /admin/login/` → 200; manual login + edit a draft + save + view list work.
  3. **Contract diff 7/7 PASS** — `pnpm contract:diff` passes for all 7 CONTRACT-04 fixtures with no allowlist additions.
  4. **Preview round-trip** — programmatically mint a token via `serve_preview()` (or use the standalone "Open preview in new tab" button — KD4-01 fallback) and load `/preview/<kind>/<token>/` end-to-end.
  5. **MinIO upload smoke** — upload an image via Wagtail admin, confirm original + renditions land in MinIO via `mc ls`, NOT on local filesystem.
- **D-VERIFY-02:** Contract-diff allowlist (`scripts/contract-diff.mjs` D-CONTRACT-02 allowlist) is **frozen**. Any new allowlist entry surfaces a 7.4 contract regression and triggers D-BUMP-03 evaluation, not an allowlist patch.
- **D-VERIFY-03:** Force-en audit row for Phase 4.1 is appended to `docs/force-en-audit.md` regardless of findings (keeps the doc's per-phase ledger pattern intact, mirrors P4). Audit walks: Wagtail admin under DevTools Sensors `en-US`, all 4 REST endpoints scanned for English month/day literals, `/preview/<token>/` view-source `<html lang="uk">`, FE homepage Ukrainian.

### Approach & Execution

- **D-APPROACH-01:** Work happens on a feature branch `chore/wagtail-7.4-lts` (NOT directly on master). Each gate commit is atomic on the branch; on full pass, the branch merges as a single PR. On D-BUMP-03 rollback, the branch is abandoned without polluting master history.
- **D-APPROACH-02:** No throwaway-branch pre-flight. The feature-branch flow itself IS the pre-flight: commit pin+lock, attempt build, iterate. The single-author/single-VPS context makes a separate spike branch unnecessary overhead.
- **D-APPROACH-03:** Task 0 (pre-implementation) is a release-notes scan: read Wagtail 7.4 LTS release notes + upgrade considerations + `wagtail-headless-preview` 0.8 compatibility matrix. If anything material to D-PIN-01..04 surfaces (deprecated APIs touched by P4 code, settings shape changes, REST API v2 serializer changes, Page-model migration requirements), the planner flags them as known risks before any code changes. This is research, not a separate plan — it's the first task of the single PLAN.

### Plan Granularity

- **D-PLAN-01:** Single PLAN.md (`04.1-01-PLAN.md`), multi-task. Tasks map 1:1 to gates plus a Task 0 (release-notes scan) and a Task 6 (ROADMAP tick + KD4-06 closure + force-en row append + Phase 4.1 SUMMARY). 7 tasks total. Each task is one atomic commit. This trades ~5 commits over a single "do everything" commit for better bisect granularity if a gate regresses post-merge.

### Ordering

- **D-ORDER-01:** Phase 4.1 runs **before** Phase 5. Rationale: P5 (single-VPS Docker deploy) ships the production stack; deploying once on 7.3 then bumping in prod doubles the verification surface. Bump in dev first, then deploy LTS. Resolves the STATE.md "user decides" open question.

### Rollback (D-BUMP-03 — locked, restated for executor)

- **D-ROLLBACK-01:** Trigger: any single gate cannot be made green within ~1 day of focused effort, OR aggregate gate-fix effort across all gates exceeds ~1 day. Action: `git revert` the pin commit, abandon the feature branch, file the blocker as Phase 4.1 known debt (KD4.1-01) with the symptom + Wagtail 7.4 release-notes pointer, and close Phase 4.1 under option C (mirroring P3/P4 pattern). KD4-06 stays open and rolls into milestone-end review.

### Claude's Discretion

- Exact wording of gate-commit messages (follow `chore(04.1): …` style, lowercase imperative, reference REQ-IDs only if a regression touches one).
- PR description format if the user opts to push the branch and review via PR (default: direct merge — single-author repo).
- Release-notes scan depth — read changelog + breaking-changes section minimum; deeper if Task 0 surfaces anything ambiguous.
- Whether to include `wagtail-headless-preview` compatibility note as inline doc comment in `pyproject.toml` (default: yes if a re-pin happened, no otherwise).
