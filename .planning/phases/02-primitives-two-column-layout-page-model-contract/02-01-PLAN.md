  <action>
1. **Install `eslint-plugin-boundaries`** as a devDependency: `pnpm add -D eslint-plugin-boundaries`. Pin the major version available at install time; record the resolved version in the SUMMARY.

2. **Extend `eslint.config.js`** by appending a new config block to the existing `tseslint.config(...)` array. The block must define element types and the boundaries rule. Per D-LIB-01 use these element-type names:
   - `core-ui-public` — files matching `projects/core-ui/src/public-api.ts`
   - `core-ui-internal` — files matching `projects/core-ui/src/lib/**`
   - `app` — files matching `src/app/**`
   - `content-models` — files matching `src/content/**`
   - `app-lib` — files matching `src/lib/**`

   Required rule (exact policy):
   ```js
   {
     files: ['**/*.ts'],
     plugins: { boundaries: boundariesPlugin },
     settings: {
       'boundaries/elements': [
         { type: 'core-ui-public',   pattern: 'projects/core-ui/src/public-api.ts' },
         { type: 'core-ui-internal', pattern: 'projects/core-ui/src/lib/**' },
         { type: 'app',              pattern: 'src/app/**' },
         { type: 'content-models',   pattern: 'src/content/**' },
         { type: 'app-lib',          pattern: 'src/lib/**' },
       ],
     },
     rules: {
       'boundaries/element-types': ['error', {
         default: 'disallow',
         rules: [
           // app may import the public API, content models, and app-lib utilities; nothing else.
           { from: 'app',              allow: ['core-ui-public', 'content-models', 'app-lib'] },
           // core-ui-internal may import other core-ui-internal files (sibling primitives + measure utils) but never `@arduino/core-ui` itself (no self-public reach).
           { from: 'core-ui-internal', allow: ['core-ui-internal'] },
           // public-api is just the barrel; it re-exports from core-ui-internal only.
           { from: 'core-ui-public',   allow: ['core-ui-internal'] },
           // content-models is leaf — no in-repo imports allowed (only TS + std lib).
           { from: 'content-models',   allow: [] },
           // app-lib is leaf — no in-repo imports allowed (matches src/lib/intl.ts pattern).
           { from: 'app-lib',          allow: [] },
         ],
       }],
     },
   }
   ```
   Import the plugin at the top of `eslint.config.js`: `import boundariesPlugin from 'eslint-plugin-boundaries';`.

3. **Override `@angular-eslint/component-selector` for `projects/core-ui/**`** so the `ui-` prefix passes lint (per PATTERNS.md). Add a second config block:
   ```js
   {
     files: ['projects/core-ui/**/*.ts'],
     rules: {
       '@angular-eslint/component-selector': ['error', { type: 'element', prefix: 'ui', style: 'kebab-case' }],
     },
   }
   ```

4. **Add a synthetic violation fixture** (matches the P1 force-en-audit "synthetic violation fixture" idiom from STATE.md) at `projects/core-ui/src/lib/__violation__.ts.example` containing a deliberate reach-through import:

   The `.example` extension keeps it out of the lint pass; the README/SUMMARY documents how to rename to `.ts` to verify enforcement once primitives exist.

5. **Run `pnpm lint`** to confirm the existing codebase still passes after the new config blocks.
